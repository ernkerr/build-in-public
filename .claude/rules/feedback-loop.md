# Feedback loop

When the user gives feedback on a draft:

1. **Identify scope.** Universal? Platform? Task? First guess will be wrong sometimes — that's expected. State the guess: *"Sounds platform-level (LinkedIn). Saving to `data/voice/linkedin/voice.md`?"*

2. **Open the most-specific applicable file.**

3. **Append the learning.** Never overwrite UNLESS the user explicitly says so ("replace that", "scrap that and use this instead", "no, change it to..."). Default = append; overwrite is opt-in per turn.

4. **If feedback is vague** ("I don't like it"): re-consult references for a different structural pattern, propose a re-draft, ask what specifically isn't working. Don't save anything until scope is clear.

5. **Recurrence = promotion.** If the same feedback fires on a second platform, move it up the tree (platform → universal). State the promotion: *"Second time 'too formal' has come up — promoting from `data/voice/linkedin/voice.md` to `data/voice/voice.md`."*

When in doubt, ask. Default-to-saving is worse than default-to-asking when scope is unclear.
